# /auto-dev-resume — 恢复中断的流程

恢复上次中断的 auto-dev 流程。

---

## 执行逻辑

### 1. 检测进度文件

检查 `.progress/status.md` 是否存在：
- 不存在 → 提示用户「没有检测到未完成的任务。请使用 /auto-dev 开始新任务。」
- 存在 → 读取状态信息

### 2. 解析当前状态

从 `.progress/status.md` 中提取：
- **任务描述**：用户的原始需求
- **当前状态**：进行中 / 已暂停
- **当前阶段**：Phase A / B / C / D
- **测试进度**：通过数 / 总数 / 自愈轮次
- **修改记录**：每轮的修改内容

### 3. 向用户报告

输出恢复摘要：

```
## 检测到未完成的任务

### 任务描述
<原始需求>

### 中断位置
- 阶段: Phase <X>
- 状态: <进行中/已暂停>
- 测试通过: X / N
- 自愈轮次: Z

### 可选操作
1. **继续** — 从中断位置继续执行
2. **重新开始** — 清除进度，从头开始
3. **查看详情** — 查看完整进度记录
```

等待用户选择。

### 4. 执行恢复

根据用户选择和中断阶段：

| 中断阶段 | 恢复动作 |
|----------|---------|
| Phase A (设计中) | 重新执行 `/auto-dev-design`（设计生成成本低，重做更安全） |
| Phase A (设计已审批，spec 生成中) | 读取已冻结的设计文档，重新执行 `/auto-dev-spec` |
| Phase B | 重新展示 test-specs，请用户审批 |
| Phase C (自愈循环中) | 校验冻结文件 → 从上次轮次继续自愈循环 |
| Phase C (已暂停) | 展示暂停诊断报告，请用户决策 |
| Phase D | 重新生成验收报告 |

### 5. 重新开始

如果用户选择重新开始：
- 归档 `.progress/status.md` 为 `.progress/status.<timestamp>.md`
- 清除 `.auto-dev.yaml` 中的 `frozen_files` 和 `frozen_checksums`
- 回到 `/auto-dev` 入口流程
