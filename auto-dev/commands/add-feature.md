# /add-feature -- 新功能开发 Skill

你是一个专业的功能开发 Agent。你采用 **Design-Test-Implement** 三阶段流程：先设计方案、再定义验收测试、最后实现代码。确保每个功能都有清晰的设计文档和完备的测试覆盖。

---

## Step 0: 探测项目环境

按 auto-dev 环境探测规则执行（见 auto-dev rules 中的「环境探测规则」段落）。额外检查 `test-specs/` 目录是否存在，没有则在后续步骤中创建。

---

## Step 1: 需求理解

1. 读取用户描述的功能需求。
2. 分析对现有代码的影响范围：
   - **需要修改的文件**：列出具体文件路径和修改原因
   - **需要新增的文件**：列出文件路径和职责说明
   - **可能受影响的已有功能**：列出需要做回归测试的模块
3. 判断复杂度：

| 级别 | 标准 | 流程 |
|------|------|------|
| **M**（中等） | 涉及 3-5 个文件，逻辑清晰，无架构变更 | 简化流程（见底部说明） |
| **L**（大型） | 涉及 5-15 个文件，需要新增模块或修改接口 | 完整三阶段流程 |
| **XL**（超大） | 涉及 15+ 个文件，架构变更，跨模块影响 | 完整三阶段 + 分批实现 |

4. 向用户输出需求理解和复杂度判断，**请用户确认**。

---

## Step 2: 设计 (Phase A)

输出功能设计方案，包含以下内容：

### 2.1 接口定义
- 新增/修改的 API 接口（路由、请求参数、响应格式）
- 新增/修改的内部函数签名
- 新增/修改的 CLI 命令（如果适用）

### 2.2 数据结构
- 新增/修改的类型、结构体、模型
- 数据库 schema 变更（如果适用）
- 配置项变更

### 2.3 核心流程
- 主要业务逻辑的流程描述
- 关键决策点和分支逻辑
- 异步/并发处理（如果适用）

### 2.4 配置项
- 新增的配置参数、默认值、说明

### 2.5 错误处理
- 可能的错误场景和处理策略
- 错误码定义（如果适用）

### 2.6 对现有代码的改动点
- 逐个文件标明：改什么、为什么改

**请用户审查设计方案**，确认后进入下一步。

---

## Step 3: 验证方案 (Phase B)

### 3.1 提取验收用例

从设计文档中提取该功能的验收用例清单，每个用例包含：
- 用例名称
- 前置条件
- 输入
- 期望输出
- 边界情况

### 3.2 生成测试规格

生成或追加 `test-specs/<feature-name>.yaml`，格式与 `/auto-dev-spec` 一致：

```yaml
spec_id: FEAT-XXX
title: <功能名称>
module: <模块路径>
approved: false

cases:
  - id: TC-001
    name: "正常流程_基本功能"
    description: "..."
    preconditions: ["..."]
    input:
      key: "value"
    expect: ["..."]
    priority: P0
  - id: TC-002
    name: "边界情况_空输入"
    description: "..."
    input: {}
    expect: ["..."]
    priority: P1
```

### 3.3 补充集成验证（如果适用）

如果存在 `scripts/verify.sh`，追加新功能的检查项。

### 3.4 用户审查

**请用户审查验收用例并冻结**。冻结后测试规格不可修改，作为实现阶段的验收标准。

冻结时计算 SHA-256 校验和，写入 `.auto-dev.yaml` 的 `frozen_checksums`。

---

## Step 4: 实现 (Phase C -- 全自动)

此阶段全自动执行，无需用户干预（除非遇到无法自愈的问题）。

### 4.1 Agent T：生成测试代码

使用独立 Agent 从 `test-specs/<feature-name>.yaml` 生成测试代码：

- **只读取接口定义和测试规格，不读取实现代码**
- 根据项目语言生成对应格式的测试文件：

| 语言 | 测试文件命名 | 放置位置 |
|------|-------------|---------|
| Go | `*_test.go` | 与被测代码同目录 |
| Python | `test_*.py` | `tests/` 目录 |
| TypeScript | `*.test.ts` | `__tests__/` 目录 |
| Rust | `#[cfg(test)]` 模块 | 被测文件内或 `tests/` 目录 |
| Java | `*Test.java` | 对应的 test 源码目录 |

### 4.2 Agent C：实现功能代码

按照设计方案实现功能代码：
- 遵循项目已有的代码风格和架构模式
- 每个文件的改动都对应设计方案中的改动点

### 4.3 自愈循环（最多 5 轮）

```
for round in 1..5:
    1. 运行新功能的测试
       - 全部 PASS → 进入回归测试
       - 有 FAIL → 分析错误，修改实现代码（不改测试），进入下一轮
```

如果 5 轮后仍有测试未通过：
- 停止实现
- 向用户输出每轮的尝试和失败原因
- 请求用户介入

### 4.4 全量回归测试

所有新功能测试通过后：

1. 如果项目有 `.auto-dev.yaml` 且配置了 `commands.test` → 运行全量测试。
2. 否则 → 运行探测到的默认测试命令。
3. 回归测试失败 → 分析是否是新功能导致的，尝试修复（仍在 5 轮限制内）。

### 4.5 集成验证（如果适用）

1. 如果存在 `scripts/verify.sh` → 运行集成验证。
2. 如果项目有容器环境（docker-compose.yaml / Dockerfile）→ 启动容器并运行集成测试。

---

## Step 5: 输出验收报告

```
## 功能验收报告

### 功能概述
- 功能名称：<名称>
- 复杂度：M / L / XL

### 文件变更
#### 新增文件
- `path/to/new_file.go` — <职责说明>

#### 修改文件
- `path/to/existing_file.go` — <改动说明>

### 测试结果
- 新功能测试：PASS（XX 个用例全部通过）
- 回归测试：PASS（XX 个测试全部通过）
- 集成验证：PASS / 跳过

### 冻结文件校验
- SHA-256 校验: 通过 ✓

### 注意事项
- <部署注意事项、配置变更、数据库迁移等>
```

## M 级别简化流程

当复杂度判断为 M（涉及 3-5 个文件，逻辑清晰）时，跳过正式文档阶段，采用简化流程：

### 简化 Step 2: 方案确认（替代 Phase A）
- 不输出正式设计文档
- 直接在对话中简要描述方案：要改哪些文件、怎么改、新增什么
- **请用户确认方案**

### 简化 Step 3: 直接写测试（替代 Phase B）
- 跳过 `test-specs/*.yaml` 的生成
- 直接编写测试代码覆盖核心场景和关键边界情况
- 运行测试，确认当前全部 FAIL（功能尚未实现）

### 简化 Step 4: 实现 + 验证
- 实现功能代码
- 运行测试 → 自愈循环（最多 5 轮）
- 运行全量回归测试

### 简化 Step 5: 简要报告
- 列出改动文件和测试结果即可，不需要完整的验收报告格式

---

## 重要原则

- **设计先行**：L/XL 级别的功能必须有用户审查通过的设计方案，不可跳过。
- **测试驱动**：先写测试，后写实现。测试是验收标准，实现阶段不可修改测试。
- **最小影响**：尽量减少对已有代码的改动，优先通过扩展而非修改来添加功能。
- **用户在环路中**：需求理解、设计方案、验收用例三个节点都需要用户确认。实现阶段全自动。
- **自愈有上限**（5 轮），超过就停下来寻求帮助，不要无限循环。
- **回归安全**：每次实现完成后必须运行全量测试，确保不破坏已有功能。
