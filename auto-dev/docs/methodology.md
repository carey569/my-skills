# auto-dev 方法论

## 问题定义

### AI 自测盲区

当 AI 既写代码又写测试时，存在根本性缺陷：

1. **确认偏差**：AI 倾向于写能通过自己代码的测试，而非真正验证需求的测试。测试和实现共享同一个"心智模型"，错误会在两端同时出现。
2. **模拟 vs 真实**：AI 在沙箱环境中无法触及真实数据库、消息队列、外部 API。单纯的 mock 测试可能全部通过，但部署后立即失败。
3. **无终止判据**：没有外部约束时，AI 可能无限循环修改测试和代码直到"看起来通过"，但实际上偏离了原始需求。

### 解决思路

将 **"谁来定义正确"** 和 **"谁来执行"** 分离：

- 人类定义"什么是对的"（审批 test-spec 和 verify.sh）
- AI 负责"怎么做到"（生成测试代码、编写实现、自动修复）
- 冻结边界保证验收标准在执行阶段不可篡改

## 四阶段生命周期

### Phase A: 需求分析、环境探测与设计方案

**目标**：理解需求，评估复杂度，生成设计方案。

1. 读取用户需求描述
2. 探测项目环境（语言、框架、基础设施），生成或读取 `.auto-dev.yaml`
3. 评估任务复杂度（S/M/L/XL）
4. 对于 L/XL 任务：生成 `designs/<feature>.md` 设计文档，经人审批后冻结
5. 从已审批的设计方案生成 `test-specs/<module>.yaml` 文件

**输出物**：
- `.auto-dev.yaml`（项目配置）
- `designs/*.md`（设计方案，待审批 → 审批后冻结）
- `test-specs/*.yaml`（测试规格，待审批）
- `scripts/verify.sh`（集成验证脚本，待审批）

**设计方案的意义**：设计方案是 test-spec 的上游输入。没有经过审批的设计，test-spec 的接口定义、数据结构、错误场景都缺乏依据。设计审批让人类在架构层面参与决策，而不仅仅审批测试用例。

### Phase B: 人工审批

**目标**：人类审查并确认验收标准。

审批分两轮：
1. **设计方案审批**（在 Phase A 中完成）：确认接口设计、数据结构、核心流程是否合理
2. **测试规格审批**：确认 test-spec 和 verify.sh 是否准确反映设计方案

测试规格审批清单：
- [ ] test-spec 中的测试用例是否覆盖设计方案中的核心场景
- [ ] 边界条件和异常路径是否考虑充分
- [ ] verify.sh 的验证步骤是否合理
- [ ] 预期断言是否准确反映设计方案中的接口契约

审批后：
- test-spec 标记 `approved: true`
- 相关文件（设计文档 + test-specs + verify.sh）加入 `frozen_files` 列表
- 计算 SHA-256 校验和，写入 `frozen_checksums`
- 进入全自动阶段

**S/M 级别任务**：简化审批流程，设计可在对话中简要描述确认，直接确认测试思路即可。

### Phase C: 全自动实现与验证

**目标**：AI 全自动完成实现和验证，无需人工介入。

执行流程：
1. 校验冻结文件完整性（SHA-256）
2. Agent T 根据冻结的 test-spec 生成测试代码
3. Agent C 根据需求和接口定义编写实现代码
4. 运行测试 -> 失败则进入自愈循环
5. 全量测试通过后运行 verify.sh
6. 再次校验冻结文件完整性
7. 所有验证通过 -> 进入 Phase D

**进度恢复**：Phase C 的每一步都记录到 `.progress/status.md`。如果中断，可通过 `/auto-dev-resume` 从中断位置继续。

### Phase D: 验收报告

**目标**：以最少的人工参与完成交付确认。

输出验收报告：
- 实现摘要（改了什么，为什么这样改）
- 测试结果（通过率、覆盖情况）
- 冻结文件校验结果
- 回归风险评估
- 待人工确认的事项（如有）

## 测试分离原则

### Agent T（测试 Agent）

**职责**：根据 test-spec 生成测试代码。

输入：
- `test-specs/*.yaml`（已审批的测试规格）
- 相关模块的**接口定义**（函数签名、类型声明、API 契约）
- **不接收实现代码**

约束：
- 只能读取接口文件，不能读取实现文件
- 生成的测试必须能独立编译/运行（允许在实现缺失时编译失败，但测试逻辑本身必须正确）
- 测试文件命名遵循语言惯例

### Agent C（实现 Agent）

**职责**：编写实现代码，使所有测试通过。

输入：
- 需求描述
- 接口定义
- Agent T 生成的测试代码（只读）

约束：
- **不可修改测试代码**（frozen_files 约束）
- 只能修改实现文件
- 修改范围应最小化

### 为什么要分离？

| 模式 | 问题 |
|------|------|
| 同一个 Agent 写测试和实现 | 测试会迁就实现，失去验证意义 |
| 人写测试，AI 写实现 | 可行但人工成本高 |
| AI 写测试规格 -> 人审批 -> AI 写测试 -> AI 写实现 | 人只审批规格（成本低），测试和实现独立（质量高） |

auto-dev 采用第三种模式。

## 冻结边界

冻结是整个方法论的核心安全机制。

### 哪些文件会被冻结？

1. 人审批后的 `designs/*.md`（设计方案）
2. 人审批后的 `test-specs/*.yaml`（测试规格）
3. Agent T 生成的测试文件
4. `scripts/verify.sh`

### 冻结意味着什么？

- Agent C 在实现阶段**不可修改**这些文件
- 如果测试失败，只能修改实现代码，不能调整测试
- 如果确实需要改测试（如发现 test-spec 本身有误），必须退回 Phase B 重新审批

### 技术强制：SHA-256 校验

v0.2.0 起，冻结不再仅靠 AI 自律，而是通过 SHA-256 校验和强制保障：

```yaml
# .auto-dev.yaml
frozen_files:
  - designs/user-service.md
  - test-specs/user-service.yaml
  - internal/user/service_test.go
  - scripts/verify.sh

frozen_checksums:
  "designs/user-service.md": "x1y2z3w4..."
  "test-specs/user-service.yaml": "a1b2c3d4..."
  "internal/user/service_test.go": "e5f6g7h8..."
  "scripts/verify.sh": "i9j0k1l2..."
```

校验时机：
- Phase C 入口（`/auto-dev-run` 开始前）
- Phase C 出口（生成报告前）
- 每 2 轮自愈循环
- verify.sh 执行时（Phase 0 步骤）

## 自愈循环

当测试失败时，Agent C 自动分析失败原因并尝试修复。

```
                +-------------------+
                |  运行测试          |
                +-------------------+
                        |
                   PASS |  FAIL
                   |    |
                   v    v
              +------+ +-------------------+
              | 完成 | | 分析失败原因       |
              +------+ +-------------------+
                              |
                              v
                        +-------------------+
                        | 修改实现代码       |
                        +-------------------+
                              |
                              v
                        +-------------------+
                        | 轮次 < 上限？      |
                        +-------------------+
                          YES |  NO
                           |    |
                           v    v
                   [回到运行测试] [暂停，报告给人]
```

### 上限设置

| 类型 | 默认上限 | 配置项 |
|------|---------|--------|
| 单元测试 | 5 轮 | `policy.unit_max_retries` |
| 集成测试 | 3 轮 | `policy.integration_max_retries` |

超限后 auto-dev 停止执行，输出：
- 每轮的修改内容和失败原因
- 建议的人工介入方向
- 当前代码状态（可编译/不可编译）

## 进度恢复

v0.2.0 新增进度恢复机制，解决"中途退出会话后无法继续"的问题。

### 进度文件

`.progress/status.md` 记录：
- 任务描述（用户原始需求）
- 当前状态和阶段
- 测试进度
- 每轮自愈的修改记录

### 恢复流程

1. `/auto-dev` 入口自动检测 `.progress/status.md`
2. 如果存在未完成任务 → 提示用户选择继续或重新开始
3. `/auto-dev-resume` 可显式触发恢复

### 各阶段恢复策略

| 中断阶段 | 恢复策略 |
|----------|---------|
| Phase A (设计中) | 重新生成设计方案（成本低） |
| Phase A (设计已审批，spec 生成中) | 读取已冻结设计，重新生成 test-specs |
| Phase B | 重新展示 test-specs 请审批 |
| Phase C | 校验冻结文件 → 从中断轮次继续 |
| Phase D | 重新生成报告 |

## 测试注入机制

解决"AI 无法访问真实外部服务"的问题。

### 策略

1. **单元测试**：使用语言原生的 mock/stub 机制，验证业务逻辑正确性
2. **集成测试**：通过 docker-compose 启动真实依赖（数据库、缓存等），在本地环境跑端到端流程
3. **verify.sh**：脚本可以调用 curl/grpcurl 等工具验证实际端点，检查数据库记录

### verify.sh 的角色

verify.sh 是 "最后一道防线"，弥补单元测试无法覆盖的场景：

```bash
#!/bin/bash
set -euo pipefail

# Phase 0: 冻结文件校验
# ... (SHA-256 校验)

# 启动服务
docker-compose up -d
sleep 3

# 验证 HTTP 端点
curl -sf http://localhost:8080/health | grep -q "ok"

# 验证业务逻辑
RESP=$(curl -sf -X POST http://localhost:8080/api/users \
  -H "Content-Type: application/json" \
  -d '{"name":"test"}')
echo "$RESP" | grep -q '"id"'

# 验证数据库记录
docker-compose exec -T db mysql -e "SELECT COUNT(*) FROM users" | grep -q "1"

# 清理
docker-compose down
echo "All verifications passed"
```

## 终止与升级条件

### 正常终止

所有条件满足时正常完成：
- 全部单元测试通过
- 全部集成测试通过（如有）
- verify.sh 通过（如有）
- 无编译/lint 错误
- 冻结文件校验通过

### 异常终止（需人工介入）

| 情况 | 处理 |
|------|------|
| 自愈循环超限 | 暂停，输出失败报告 |
| test-spec 本身有误 | 退回 Phase B 重新审批 |
| 环境问题（docker 启动失败等） | 暂停，请用户检查环境 |
| 需求歧义 | 暂停，请用户澄清 |
| 冻结文件被篡改 | 立即暂停，报告校验失败 |

### 升级路径

当 Agent 无法自行解决时：
1. 输出当前状态和失败原因
2. 列出已尝试的方案
3. 提供 2-3 个建议的人工介入方向
4. 等待用户指令后继续
