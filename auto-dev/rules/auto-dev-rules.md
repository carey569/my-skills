## auto-dev rules

### 任务复杂度分级

| 级别 | 标准 | 流程 |
|------|------|------|
| **S** | 单文件改动，逻辑简单 | 直接实现 → 跑已有测试 |
| **M** | 2-3 文件，逻辑清晰 | 先写测试 → 再写实现 → 验证 |
| **L** | 4+ 文件或跨模块 | 生成 test-spec → 人审批 → Agent T 写测试 → Agent I 实现 → 验证 |
| **XL** | 架构变更或新子系统 | 同 L，增加设计评审和分阶段交付 |

### 核心规则

1. **测试与实现分离**：L/XL 任务使用独立 Agent 分别负责测试和实现，避免球员兼裁判。
2. **冻结边界**：用户审批过的 `test-specs/*.yaml`、`verify.sh`、Agent T 生成的测试文件标记为冻结，实现阶段不可修改。
3. **自愈循环上限**：单元测试最多重试 5 轮，集成测试最多 3 轮，超限暂停并向用户报告失败原因。
4. **先理解再动手**：修改任何代码前，必须先读取相关文件理解上下文。
5. **错误脱敏**：面向用户的错误信息不得暴露内部路径、堆栈或敏感配置。

### 可用技能

- `/auto-dev` — 自动开发全流程（init / spec / implement / verify）
- `/fix-bug` — 定位并修复 bug
- `/add-feature` — 添加新功能
